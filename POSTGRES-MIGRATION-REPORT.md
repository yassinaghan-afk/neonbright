## 2. Models and indexes (prisma/schema.prisma)

| Model | Notes | Key indexes / constraints |
|---|---|---|
| `PortfolioCategory` | Events, Brands, etc. | `@id`, `slug @unique`, `(enabled, sortOrder)` |
| `PortfolioProject` | Events + Brands share this via `categoryId` | `@id`, `@@unique(categoryId, slug)`, `slug`, `categoryId`, `published`, `sortOrder`, `(categoryId, published, sortOrder)` |

| `SiteSetting` | key/value JSONB — one row per singleton block | `key @id` |
| `CmsRevision` | audit/optimistic-concurrency trail | `createdAt` index |

## 3. Fields stored as JSONB and why

- `SiteSetting.value` — singleton copy blocks (`hero`, `sectionCopy`, `instagram` settings, `company`, `contact`, `social`, `seo`, `heroMediaVersion`, deprecated `legacyProjects`). These are single flexible objects, not collections; one row **per block**, never the whole CMS in one row.
- `custom` (nullable JSONB on every collection model) — any unknown/extra key found on a JSON record is preserved here and merged back on read, so custom fields survive round-trips. Proven by test.
- Everything queryable (ids, slugs, titles, descriptions, published, sortOrder, media URLs, tags…) is a real column with indexes.

## 5. JSON → PostgreSQL mapping

- Every record keeps its **exact existing string ID** as the primary key.
- Slugs, categoryIds, URLs, published/enabled, sortOrder copied verbatim.
- `images[]`/`gallery[]`/`videos[]` become ordered `ProjectMedia` rows (`role` + `sortOrder`) and are reassembled into identical arrays on read.
- `partners` and `brandsPageLogos` share the `Partner` table via `kind`.
- Instagram `createdAt`/`updatedAt` ISO strings become timestamps and are serialized back to ISO strings on read.
- Unknown keys on any record → `custom` JSONB, merged back on read.

## 6. Import script behavior

`scripts/import-cms-json-to-postgres.ts`:
1. Reads `$STORAGE_ROOT/cms-content.json` (override with `CMS_JSON_PATH`)
2. Validates: JSON parse, array shapes, project→category references, duplicate IDs
3. Prints per-collection counts (JSON vs DB comparison table)
4. **Dry-run by default** — `--apply` required to write
5. Before apply: timestamped backup to `$STORAGE_ROOT/backups/cms-content-pre-pg-import-<ts>.json`
6. Single `$transaction` — one failed record rolls back everything
7. Idempotent upserts by stable ID — rerun never duplicates (tested)
8. Never deletes existing database records
9. Post-import verify: counts + IDs + slugs + categoryId + published + sortOrder

## 7. Phased cutover

- `CMS_STORAGE=json` (default, current production) — JSON store untouched.
- `CMS_STORAGE=compare` — JSON serves everything; PostgreSQL read in the background per request; count mismatches logged as `[cms-compare]` without content values; PostgreSQL is never written outside the import script.
- `CMS_STORAGE=postgres` — reads and Admin writes go to PostgreSQL in transactions via the existing `readCMSContent`/`updateCMSContent` entry points, so **all existing Admin routes, safe-update helpers, and revalidation logic work unchanged**. No dual writes; JSON becomes rollback data.

Admin CRUD in postgres mode: routes still build the full merged content (partial-update safety via the existing `safeUpdate*` helpers), then `postgres-writer` upserts by ID and deletes only records explicitly removed from a collection, inside one transaction; a `CmsRevision` row records every mutation; `revalidatePublicSite()` runs only after commit.

## 9. Remaining differences / limitations

- In `postgres` mode the JSON-only conveniences (`ensureBrandsPageProjects` seed repair, hero-media auto-sync) don't run on read — data is what the import/Admin wrote. This is intentional: the DB is authoritative.
- Compare mode diff-logs collection counts (projects, testimonials, reviews, instagram, hero, partners); it is a canary, not a byte-differ.
- Optimistic-concurrency 409 responses still use the CMS `revision` field carried through `CmsRevision`; same behavior as JSON mode.
- Docker image build with the new entrypoint was not run here (no Docker on this machine) — Dockerfile changes are additive and mirror the tested local flow (`prisma generate` → build; `migrate deploy` → start).
